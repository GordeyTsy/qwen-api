# Анализ процесса запуска и деплоя `qwen-api`

Этот документ описывает, как устроена архитектура деплоя и процесс запуска модели в среде Kaggle для проекта `qwen-api`.

## 1. Подготовка и деплой (Push)

Процесс деплоя инициируется локально с помощью скрипта `scripts/push_kaggle_notebook.sh`.

### Что делает `push_kaggle_notebook.sh`:
*   **Проверки статуса**: Скрипт сначала опрашивает Kaggle API (`kaggle kernels status`), чтобы убедиться, что целевой ноутбук (`gordeytsy/qwen-api`) или старые/устаревшие версии (legacy) в данный момент не запущены. Это необходимо для экономии квот (Quota Discipline).
*   **Флаг `--force`**: По умолчанию скрипт прервет работу, если найдет активный процесс. Использование флага `--force` позволяет игнорировать активные сессии и принудительно обновить ноутбук (однако старую сессию придется останавливать вручную через UI или API контроля).
*   **Публикация (Push)**: Выполняет команду `kaggle kernels push` из директории `kaggle/`, явно указывая акселератор `--accelerator NvidiaTeslaT4`.

### Требования к метаданным:
Файл `kaggle/kernel-metadata.json` намеренно оставлен почти пустым. В нем отключено монтирование датасетов (`"dataset_sources": []`) и моделей (`"model_sources": []`). Это сделано для того, чтобы избежать долгих задержек (pre-cell mounting delays) на стороне Kaggle при запуске сессии.

## 2. Среда исполнения Kaggle (Ноутбук)

Сам ноутбук `kaggle/qwen-api.ipynb` максимально минималистичен и выполняет роль простого загрузчика (bootstrapper).

### Шаги внутри ноутбука:
1.  **Настройка окружения**: В ячейке жестко прописаны ключи доступа (секреты), такие как `QWEN_API_KEY`, `QWEN_CONTROL_TOKEN`, `QWEN_LOG_TOKEN` и топик для уведомлений `QWEN_NTFY_TOPIC`. (Прим.: в реальной безопасной среде они могут подтягиваться из Kaggle Secrets).
2.  **Клонирование репозитория**: Клонирует публичный репозиторий проекта из GitHub (`https://github.com/GordeyTsy/qwen-api.git`) в директорию `/kaggle/working/qwen-api`.
3.  **Установка зависимостей**: Выполняет установку скачанного пакета: `pip install -e /kaggle/working/qwen-api`.
4.  **Запуск приложения**: Вызывает главный скрипт: `python -m qwen_api.kaggle_runner`.

## 3. Runtime Bundle (Среда выполнения LLaMA)

Проект не собирает `llama-server` (движок инференса) внутри Kaggle, так как это долго. Вместо этого используется предварительно собранный пакет (Runtime Bundle).

### Как он создается:
Скрипт `scripts/package_runtime_bundle.sh` берет готовую сборку `llama-server` (с поддержкой CUDA sm75 для Tesla T4) и необходимые динамические библиотеки (`.so` файлы), упаковывает их в архив `.tar.gz`, генерирует манифест с SHA256-хэшами.
Этот архив затем публикуется как GitHub Release Asset (например, `runtime-bundle-sm75-v1`).

## 4. Процесс работы `kaggle_runner`

Хотя код самого `qwen_api/kaggle_runner.py` мы не разбирали детально, из `OPERATIONS.md` и контекста ясно, что он делает следующее:
1.  **Загрузка Runtime Bundle**: Скачивает релизный архив с `llama-server` и распаковывает его.
2.  **Загрузка весов**: Скачивает необходимые GGUF веса модели (по умолчанию `HauhauCS/Qwen3.6-35B-A3B-Uncensored-HauhauCS-Aggressive`, квантование `Q4_K_P`).
3.  **Запуск туннеля**: Поднимает Cloudflare Quick Tunnel, чтобы сделать локальный сервер доступным извне.
4.  **Запуск сервера**: Запускает `llama-server`, натравливая его на скачанные веса и направляя трафик через Cloudflare.
5.  **Обеспечение Observability**: Предоставляет URL для логов (`$LOG_URL/events`) и управления сервером (`$CONTROL_URL/stop`) через зашифрованные токены.

## Резюме
Подход очень элегантный: вместо того, чтобы держать тяжелые датасеты в Kaggle или собирать C++ код на лету, ноутбук выполняет роль легковесного контейнера. Он мгновенно стартует, сам подтягивает свежий код из GitHub (runner), затем runner динамически скачивает прекомпилированный бинарник (`llama-server`) и веса, разворачивает туннель Cloudflare и начинает обрабатывать запросы через OpenAI-совместимый API.
